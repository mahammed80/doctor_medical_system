

**وثيقة الخدمات التقنية**

منصة الاستشارات الطبية الإلكترونية — د. خالد بترجي

مُعدّ بواسطة: AI Supervision

التاريخ: 25 مايو 2026

تصف هذه الوثيقة جميع الخدمات والمكونات التقنية التي تم تطويرها وتسليمها ضمن منصة الاستشارات الطبية الإلكترونية الخاصة بالدكتور خالد بترجي، وتشمل الواجهات، قواعد البيانات، التكاملات الخارجية، ونظام التصميم.

**1\. الصفحة الرئيسية (Landing Page)**

**المسار:** /

**النوع:** صفحة ثابتة — Static Page

صفحة استقبال المريض وأول نقطة تواصل مع المنصة.

**▸  المحتوى والمكونات**

•  شريط علوي ملوّن بلون العلامة التجارية

•  عنوان رئيسي يعرض اسم الدكتور مع تمييز بصري

•  نص تعريفي بخدمة الاستشارة الإلكترونية

•  زر "ابدأ الاستشارة" يوجّه المستخدم لنموذج الحجز

•  صورة احترافية للدكتور مع بطاقة تعريفية

•  مؤشرات إحصائية (4 خطوات فقط — \+35 سنة خبرة — 100% أون لاين)

**▸  التقنيات المستخدمة**

•  Next.js App Router — Server Component

•  خط Tajawal العربي عبر Google Fonts

•  تصميم RTL كامل (يمين لليسار)

•  Next.js Image مع أولوية تحميل (priority)

**2\. نموذج الاستشارة متعدد الخطوات**

**المسار:** /consultation/new

**النوع:** Client Component — Interactive Multi-step Form

المسار الكامل لحجز الاستشارة من البداية حتى تأكيد الموعد، مقسّم على 4 خطوات.

**▸  الخطوة 1 — بيانات المريض**

•  اسم المريض كاملاً

•  رقم الجوال

•  العمر

•  سبب الاستشارة (الشكوى الرئيسية)

•  التاريخ المرضي (اختياري)

•  الأدوية الحالية (اختياري)

•  التحقق من اكتمال الحقول الإلزامية قبل الانتقال

**▸  الخطوة 2 — رفع الملفات الطبية**

•  منطقة سحب وإفلات (Drag & Drop Zone)

•  دعم الصيغ: PDF، JPG، PNG، DICOM

•  رفع الملفات مباشرة إلى Supabase Storage

•  عرض الملفات المرفوعة مع إمكانية الحذف

•  تحديث فوري للواجهة عند اكتمال الرفع

**▸  الخطوة 3 — الدفع**

•  عرض سعر الاستشارة (قابل للتخصيص عبر متغير البيئة)

•  بطاقات اختيار طريقة الدفع (بطاقة ائتمانية / Apple Pay / Mada)

•  حقول بيانات البطاقة مع تنسيق مرئي احترافي

•  تحديث حالة الاستشارة إلى "في انتظار الحجز" بعد الدفع

•  تسجيل رقم العملية (Payment ID) في قاعدة البيانات

**▸  الخطوة 4 — حجز الموعد**

•  تضمين Calendly iframe مباشرة في الصفحة

•  المريض يختار الوقت المناسب من جدول الدكتور الفعلي

•  رصد حدث calendly.event\_scheduled لتأكيد الحجز تلقائياً

•  تحديث حالة الاستشارة إلى "محجوز" وحفظ رابط الحدث

•  توجيه المريض إلى صفحة النجاح عند اكتمال الحجز

**▸  ميزات عامة للنموذج**

•  شريط تقدم مرئي يوضح الخطوة الحالية من 4

•  زر "السابق" للعودة للخطوة السابقة

•  حفظ بيانات الاستشارة في Supabase منذ الخطوة الأولى

•  دعم وضع العرض التجريبي (Demo Mode) عند غياب الإعدادات

•  Spinner تحميل أثناء العمليات غير المتزامنة

**3\. صفحة تأكيد الحجز**

**المسار:** /consultation/success

**النوع:** Server Component — Static

صفحة نهائية تُعرض للمريض عقب اكتمال الحجز.

•  أيقونة نجاح دائرية مع تأثير لوني

•  رسالة تأكيد باسم الدكتور

•  إشعار بإرسال إيميل يحتوي تفاصيل الجلسة ورابطها

•  زر للعودة إلى الصفحة الرئيسية

**4\. لوحة تحكم الدكتور**

**المسار:** /dashboard

**النوع:** Server Component — Dynamic (force-dynamic)

واجهة إدارة خاصة بالدكتور لاستعراض جميع الاستشارات الواردة.

**▸  عداد الحالات**

•  "في انتظار الدفع" — مميز بالأصفر

•  "في انتظار الحجز" — مميز بالأزرق

•  "محجوز" — مميز بالأخضر

**▸  جدول الاستشارات**

•  اسم المريض

•  رقم الجوال

•  سبب الاستشارة (مختصر)

•  الحالة الحالية بشارة ملونة

•  تاريخ إنشاء الاستشارة

•  رابط "عرض" للانتقال لتفاصيل كل استشارة

•  ترتيب تنازلي حسب تاريخ الإنشاء (الأحدث أولاً)

•  رسالة فارغة عند عدم وجود استشارات

**5\. صفحة تفاصيل الاستشارة**

**المسار:** /dashboard/\[id\]

**النوع:** Server Component — Dynamic

عرض كامل لبيانات استشارة واحدة مع ملفاتها الطبية.

**▸  بيانات المريض**

•  الاسم الكامل

•  رقم الجوال

•  العمر

•  سبب الاستشارة

•  التاريخ المرضي

•  الأدوية الحالية

•  تاريخ ووقت تقديم الاستشارة

**▸  الملفات الطبية**

•  قائمة بجميع الملفات المرفوعة

•  اسم كل ملف مع رابط مباشر لفتحه في تبويب جديد

•  رسالة توضيحية عند عدم وجود ملفات

•  زر "← لوحة التحكم" للعودة

•  معالجة حالة الاستشارة غير الموجودة (404 notFound)

**6\. قاعدة البيانات (Supabase PostgreSQL)**

البنية الكاملة لقاعدة البيانات المستخدمة في تخزين بيانات الاستشارات والملفات.

**▸  جدول consultations — الاستشارات**

•  id — UUID مفتاح رئيسي (auto-generated)

•  patient\_name — اسم المريض

•  patient\_phone — رقم الجوال

•  patient\_age — العمر

•  chief\_complaint — سبب الاستشارة

•  medical\_history — التاريخ المرضي (اختياري)

•  current\_medications — الأدوية الحالية (اختياري)

•  status — الحالة: pending\_payment | pending\_booking | booked

•  payment\_id — رقم عملية الدفع (اختياري)

•  calendly\_event\_url — رابط حدث Calendly (اختياري)

•  created\_at — تاريخ ووقت الإنشاء (timestamptz)

**▸  جدول consultation\_files — الملفات الطبية**

•  id — UUID مفتاح رئيسي

•  consultation\_id — مفتاح خارجي مرتبط بجدول الاستشارات (Cascade Delete)

•  file\_name — اسم الملف الأصلي

•  file\_url — رابط الملف في التخزين السحابي

•  file\_type — نوع/امتداد الملف

•  created\_at — تاريخ الرفع

**▸  إعدادات الأمان (Row Level Security)**

•  تفعيل RLS على كلا الجدولين

•  سياسات مفتوحة لـ MVP — قابلة للتضييق بإضافة المصادقة لاحقاً

**7\. التخزين السحابي للملفات (Supabase Storage)**

**اسم الـ Bucket:** consultation-files

**النوع:** Public Bucket

نظام تخزين الملفات الطبية المرفوعة من المرضى.

•  رفع الملفات مباشرة من المتصفح (Client-side Upload)

•  سياسة "allow uploads" — تسمح بالرفع لأي مستخدم

•  سياسة "allow reads" — تسمح بالقراءة العامة (public URLs)

•  الصيغ المدعومة: PDF، JPG، PNG، DICOM

•  الحجم الأقصى للملف: محدود بإعدادات Supabase الافتراضية

•  الروابط دائمة وقابلة للمشاركة المباشرة

**8\. التكامل مع Calendly**

**متغير البيئة:** NEXT\_PUBLIC\_CALENDLY\_URL

ربط جدول الدكتور الفعلي مع منصة الحجز لتمكين المريض من اختيار موعده.

•  تضمين Calendly Widget مباشرة داخل صفحة الحجز (iframe)

•  المريض يرى المواعيد المتاحة الفعلية للدكتور

•  رصد حدث calendly.event\_scheduled عبر postMessage

•  حفظ رابط الحدث في قاعدة البيانات تلقائياً

•  تحديث حالة الاستشارة إلى "محجوز" فور تأكيد الموعد

**9\. نظام التصميم**

منظومة بصرية متكاملة مصممة للغة العربية واتجاه RTL.

**▸  الألوان**

•  الأساسي (Primary) — أزرق oklch(50% 0.20 255\)

•  الذهبي (Gold) — لحالة "انتظار الدفع"

•  الأخضر (OK) — لحالة "محجوز"

•  الأحمر (Error) — للأخطاء والتحذيرات

•  نظام ألوان السطوح: bg / surface / surface-up

**▸  المكونات الجاهزة**

•  .btn-primary — زر أساسي مع hover وdisabled وspring animation

•  .btn-ghost — زر ثانوي شفاف مع حدود

•  .input — حقل إدخال مع focus ring بلون العلامة التجارية

•  .label — تسمية الحقل

•  .card — بطاقة محتوى مع ظل وحدود ناعمة

•  .field — غلاف مجموعة الحقل والتسمية

**▸  الخطوط والتايبوغرافي**

•  Tajawal — خط عربي من Google Fonts (300 → 900\)

•  Inter — للأرقام والنصوص اللاتينية

•  direction: rtl على مستوى الـ body

•  HTML lang="ar" dir="rtl"

**▸  الحركة والانيميشن**

•  ease-out: cubic-bezier(0.22, 1, 0.36, 1\)

•  ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1\)

•  spin keyframe للـ Spinner

•  hover transitions على الأزرار والصفوف

**12-أ. مستويات الاستشارة والتسعير (Consultation Levels)**

المطلوب إعادة هيكلة منظومة الاستشارة إلى مستويات تدريجية بدلاً من سعر ثابت واحد، بحيث يختار المريض المستوى المناسب لحالته.

**▸  المستوى الأول — الكشف الأساسي (889 ريال)**

•  الكشفية الأساسية مع الدكتور

•  عرض وتحليل الأشعة

•  التشخيص الطبي

•  مدة الجلسة: 30 دقيقة

•  متابعة مجانية (Follow-up) لمدة 10 أيام فقط بعد الجلسة

ملاحظة:  بعد انتهاء الـ 10 أيام تُحسب المتابعة كاستشارة جديدة بالسعر الكامل.

**▸  المستوى الثاني — الكشف الشامل مع خطة علاجية (1700 ريال)**

•  الكشفية الأساسية مع الدكتور

•  عرض وتحليل الأشعة

•  التشخيص الطبي

•  إعداد برنامج علاجي متكامل

•  تقييم الحاجة للجراحة وعرض خيار العملية على المريض

•   في حال موافقة المريض على الجراحة: تُناقَش التفاصيل ويُقدَّم عرض بتكلفة الجراحة

**12-ب. متابعة ما بعد الجراحة**

نظام متابعة مرن للمرضى الذين أجروا العملية مع الدكتور، يوفر خيارين:

**▸  الخيار الأول — دخول العيادة مجدداً كمراجعة**

•  أول مراجعة بعد العملية: مجانية تماماً (Free of Charge)

•  المراجعات التالية: بسعر الكشفية الأساسية (المستوى الأول)

**▸  الخيار الثاني — باقات الجلسات المتعددة (Packages)**

إذا احتاج المريض عدة كشفيات متكررة يُقدَّم له سعر حزمة بدلاً من الدفع المنفرد:

| الباقة | عدد الجلسات | السعر العادي | سعر الباقة |
| ----: | ----: | ----: | ----: |
| باقة 3 جلسات | 3 | 2,700 ريال | **2,500 ريال** |
| باقة 4 جلسات | 4 | 3,600 ريال | **3,400 ريال** |

**12-ج. توصيل الأدوية للمنزل (Home Medication Delivery)**

طلب الدكتور دراسة إمكانية وصول وصفة الدواء إلى المريض في منزله مباشرة بعد الاستشارة دون الحاجة للذهاب للصيدلية.

**▸  النقاط التي تحتاج دراسة**

•   آلية تسجيل بيانات المندوب وإدارة إشعاراته (SMS / واتساب / تطبيق)

•  آلية إرسال الوصفة الطبية بصورة آمنة وموثوقة (PDF موقّع — وصفة رقمية معتمدة)

•  التغطية الجغرافية: هل التوصيل داخل مدينة معينة أم أوسع؟

•  تكلفة التوصيل: هل تُدرج في فاتورة الاستشارة أم يدفعها المريض للصيدلية مباشرة؟

•  الإطار التنظيمي: اشتراطات هيئة الصحة / الدواء لوصف الدواء إلكترونياً عن بُعد

**▸  المتطلب التقني المبدئي إن أمكن التطبيق**

•  حقل أو صفحة لإدخال الوصفة داخل لوحة تحكم الدكتور

•   ‫إرسال إشعار تلقائي للمندوب المسؤول يتضمن: اسم المريض، الدواء المطلوب، وعنوان التوصيل

•  إشعار للمريض بحالة الطلب (قيد التحضير / في الطريق / تم التسليم)

**12-د. إجراء تحاليل الدم وربط المختبر (Lab Test Workflow)**

الدكتور لديه مختبر يتعامل معه، ويريد آلية واضحة تُمكّن المريض من معرفة التحاليل المطلوبة منه والذهاب لإجرائها واستقبال النتائج.

**▸  التساؤلات المطروحة من الدكتور**

•  كيف يعرف المريض أن عليه إجراء تحليل دم محدد؟

•  كيف يعرف المريض أي مختبر يذهب إليه؟

•  كيف تصل نتائج التحليل للدكتور بشكل منظم؟

•  ما إجراءات هذه العملية بالكامل (البروسيجير)؟

**▸  المقترح التقني للدراسة**

•  بعد الاستشارة: الدكتور يُصدر "طلب تحليل" من لوحة التحكم يحدد فيه التحاليل المطلوبة

•  المريض يستلم إشعاراً (SMS / إيميل) بالتحاليل المطلوبة واسم المختبر وعنوانه

•   إرسال إشعار تلقائي للمسؤول في المختبر بطلب التحليل فور إصداره من الدكتور، وإشعار تأكيد للدكتور والمريض بعد رفع النتائج

•  النتائج تظهر مباشرة في صفحة تفاصيل الاستشارة في لوحة تحكم الدكتور

•  الدكتور يستطيع مراجعة النتائج وتحديث خطة العلاج

**▸  نقاط تحتاج توضيحاً من الدكتور**

•  هل المختبر الشريك لديه نظام رقمي أو يعمل يدوياً؟

•  هل المريض يدفع تكلفة التحليل مباشرة للمختبر أم تُدرج في الاستشارة؟

•  هل النتائج يرفعها المريض بنفسه أم المختبر يرسلها مباشرة؟

جميع البنود الواردة في هذا القسم هي مقترحات أولية قيد الدراسة. يُنصح بتحديد الأولويات مع الدكتور خالد ثم إصدار أمر عمل منفصل لكل بند يُعتمد تطويره.
